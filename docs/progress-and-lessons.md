# 项目整理进度与经验记录

## 本轮目标

本轮工作的核心目标有两类：

1. 优化现有静态工具站点的首页展示和重点页面实现。
2. 调整仓库结构，让首页、工具页和公共资源的边界更清晰。

### 1. 首页 `index.html` 完善

- 新增“工具库”展示区域，不再只通过导航下拉暴露工具入口。

这一步的意义在于减少页面之间重复写同类逻辑，为后续按页面继续优化打基础。

## 当前状态总结

截至本次提交，项目已经从“根目录平铺多个 HTML 页面”的状态，整理为“首页入口 + 页面目录 + 公共资源目录”的结构。

这意味着：

- 用户进入路径更清晰
- 首页具备更完整的工具展示能力
- 工具页的存储、下载等能力开始具备统一抽取基础
- 后续如果继续优化，只需要针对具体页面逐步处理，而不必再次大规模搬迁目录

## 工作经验

### 1. 结构优化要分层推进

这次实践证明，先做目录级整理，再做公共能力抽取，比一开始就拆所有页面的 CSS/JS 更稳妥。

如果一上来同时做：

- 改目录
- 改路径
- 改脚本结构
- 改渲染逻辑

那么排查问题会非常痛苦。分阶段推进能明显降低风险。

### 2. 对静态项目来说，“统一入口”比“立即组件化”更重要

这个仓库本质上是一个多工具静态站点，而不是单页应用。

在这种前提下，最优先的不是马上引入复杂工程化，而是先解决：

- 页面放在哪里
- 首页怎么找到它们
- 公共资源怎么复用
- 用户从哪里进入

这几件事解决后，项目的可维护性已经会提升很多。

### 3. 公共工具抽取的收益很高

虽然只抽了一个 `app-utils.js`，但已经覆盖了多个页面都在重复做的事：

- 本地存储
- 防抖
- 下载
- 转义

这种抽取非常适合静态项目，投入小、收益快、风险也低。

### 4. 首页文案需要站在“访客视角”

首页最开始的问题，不是功能不够，而是很多表达像内部汇报材料，例如：

- 已入库
- 统一维护
- 静态工具页
- 测试覆盖率
- 效能数据

这些词对开发者有意义，但对访客不友好。调整为“能力概览、实践案例、工具库、项目亮点”后，页面更像对外展示页。

### 5. 目录重组后最容易出问题的是路径

本轮实际出现的问题也说明：

- HTML 文件移动后，资源路径最容易失效
- 用户未必会立即按新地址访问页面
- 页面本身没坏，不代表入口就没问题

因此后续只要再做结构调整，第一优先永远是核对：

- 首页跳转路径
- 页面资源路径
- 相对路径与根路径混用情况

## 后续建议

如果后续继续推进，建议按下面顺序做：

1. 先做一轮浏览器回归验证，重点检查首页和核心工具页是否都能正常打开、保存和导出。
2. 再按实际使用频率，优先整理 `QuotePilot`、`InvestmentDashboard`、`TripValidator` 这些体量较大的页面。
3. 当某个页面继续膨胀时，再单独把该页面的内联脚本和样式拆出去，而不是全仓统一硬拆。

## 结论

本轮已经完成了“能看、能找、能继续维护”的第一阶段整理。

相比起继续追求一次性彻底重构，当前成果更适合先稳定下来，作为后续迭代的基础版本。
